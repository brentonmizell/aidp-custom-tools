# Credentials — when each tool needs what, and how to store it

One page that every tool README in this directory links to. The goal is to
make it obvious **when** a tool needs a credential, **what kind**, and
**how to store it safely** so the same instructions don't have to live in
twelve different READMEs.

---

## TL;DR — decision tree

```
Does your tool make any network call?
├── No  ─────────────────────────────────────────── No credential needed.
│                                                   Leave conf.credential_name empty.
│                                                   Examples: text_utils_toolkit,
│                                                   python_runner_tool, convert_file_tool.
│
└── Yes
    │
    ├── It calls a public AIDP / OCI REST API
    │   (catalogs, schemas, volumes, KBs, compute,
    │    document understanding, object storage, GenAI)
    │   ──────────────────────────────────────────  Two interchangeable paths
    │                                                (conf.credential_name auto-routes):
    │                                                 • Display name → AIDP Credential
    │                                                   Store (needs new aidputils.secrets).
    │                                                 • OCID `ocid1.vaultsecret.…` →
    │                                                   OCI Vault (works on every runtime
    │                                                   with resource principal).
    │                                                See section A0 for the Vault setup
    │                                                + A for the Credential Store setup.
    │
    ├── It connects to an Oracle Database
    │   (selectai_toolkit, anything via oracledb)
    │   ──────────────────────────────────────────  Two pieces:
    │                                                  (a) DB connect creds (username,
    │                                                      password, wallet) — store as
    │                                                      SECRET_TOKEN credential.
    │                                                  (b) For DBMS_CLOUD_AI: a DBMS_CLOUD
    │                                                      credential inside the ADB itself
    │                                                      (provisioned once via SQL).
    │
    ├── It needs SMTP / email server creds
    │   (email_toolkit)
    │   ──────────────────────────────────────────  Store as SECRET_TOKEN with keys
    │                                                host / port / username / password / from.
    │
    └── It calls a third-party HTTP API with a bearer token
        (web_toolkit, REST tools generated by the wizard)
        ──────────────────────────────────────────  Store as SECRET_TOKEN with one
                                                    key per token, e.g.
                                                    api_key / refresh_token / etc.
```

## Why the Credential Store at all

A custom code tool runs inside the AIDP agent runtime. Three options for how
it authenticates outbound calls:

| Path | When it works | Why we don't recommend it for production |
|---|---|---|
| **Hardcoded PEM in the zip** | Demos only. | Anyone with the zip has the key. No rotation. Re-uploading every rotation. |
| **`~/.oci/config` shipped in the zip** | Demos only. | Same problem + relative-path resolution is fragile in the runtime. |
| **`oci.auth.signers.get_resource_principals_signer()`** | AIDP-internal calls (the ones the runtime itself makes). | **Returns 401 against the public AIDP data-plane APIs** (catalogs / schemas / volumes / KBs). Confirmed by the Jun-17 JR/Sambit thread. Until the service-side auth path is fixed, do not rely on it for those endpoints. |
| **`aidputils.secrets.get(name)` ⟶ `oci.signer.Signer(private_key_content=...)`** | Public AIDP / OCI APIs from inside a custom tool. | **This is the production pattern.** No PEM in the zip, no key in source, rotates by updating the credential without redeploying. |

**Bottom line:** if your tool calls a *public* AIDP or OCI API, use the
Credential Store path. Resource principal is enough only for AIDP-internal
calls (the ones the runtime is already wired for).

## Operator setup — how to create a credential in AIDP

You do this **once per environment**, in the AIDP console, not in the tool.
The tool never sees the raw values, only the credential's display name.

### A0. OCI Vault secret (recommended when aidputils.secrets isn't in your runtime)

The Credential Store path below relies on `aidputils.secrets.get(name)`, which
is only available in **newer** AIDP agent runtimes. If
`CUSTOM_CODE_TOOLS/runtime_probe/` reports `aidputils.secrets: NOT available`,
use this path instead — it works against any runtime with the modern OCI SDK
+ resource principal (every AIDP runtime tested as of writing).

How it works: store the same JSON bundle in **OCI Vault** instead of AIDP's
Credential Store. The tool detects an OCID-shaped `credential_name`
(starts with `ocid1.vaultsecret.`), pulls the secret bundle via
`oci.secrets.SecretsClient.get_secret_bundle()` using the agent runtime's
resource principal, base64-decodes + JSON-parses the content, and uses it
exactly like a Credential Store bundle.

**One-time setup:**

1. **OCI Console → Vault → Create Vault** (if you don't have one). Pick a
   compartment the agent runtime can read from. Standard tier is fine.
2. **In the Vault → Master Encryption Keys → Create Key** (AES-256, software
   protection mode is fine for non-FIPS).
3. **Vault → Secrets → Create Secret.**
   - Name: pick something stable, e.g. `aidp_oci_key`.
   - Encryption Key: the key from step 2.
   - **Secret Type Template:** Plain-text.
   - **Secret Contents:** paste the JSON object below, with your real values:
     ```json
     {
       "tenancy":     "ocid1.tenancy.oc1..aaaaaaaa...",
       "user":        "ocid1.user.oc1..aaaaaaaa...",
       "fingerprint": "aa:bb:cc:dd:ee:ff:11:22:33:44:55:66:77:88:99:00",
       "private_key": "-----BEGIN PRIVATE KEY-----\nMIIEvQ...\n-----END PRIVATE KEY-----\n"
     }
     ```
     (Use `\n` for line breaks inside the JSON string; or paste the literal
     multi-line PEM and ensure your JSON encoder escapes it correctly.)
4. **Save and copy the Secret OCID** — looks like
   `ocid1.vaultsecret.oc1.iad.amaaaaaa...`.
5. **Grant the agent runtime's dynamic group `read secret-bundles`** on the
   compartment containing this secret. Sample IAM policy:
   ```
   allow dynamic-group <your-agent-dg> to read secret-bundles in compartment <compartment-name>
   ```
6. **Use it:** set `conf.credential_name` to the Vault secret OCID. The
   tools auto-route on the `ocid1.vaultsecret.` prefix — no other changes.

**Bundle key naming:** the JSON keys are the same as for the AIDP Credential
Store path. For OCI API auth use `tenancy` / `user` / `fingerprint` /
`private_key`. For SMTP: `host` / `port` / `username` / `password` /
`from_address`. For DB: `username` / `password` / `connection_string` /
`wallet_b64`. The tool's `_execute_tool` reads the bundle as a dict; the key
names need to match what that specific tool expects.

**Same `credential_name` field; auto-routing on prefix.** If you later
upgrade to a runtime with `aidputils.secrets`, you can switch back to a
display name without touching the tool code — just change the value in
`conf.credential_name`. Display names go to AIDP, OCIDs go to Vault.

### A. OCI API-key credential (most common — for AIDP/OCI REST calls)

1. **Mint an OCI API key pair** under a dedicated service user (least
   privilege; only the IAM policies the tools need). Don't reuse a
   personal key.
2. **AIDP → Settings → Credentials → New** (or the workspace-scoped
   equivalent your tenancy uses).
3. **Type:** `SECRET_TOKEN` (not `SERVICE_ACCOUNT`, not `VAULT_REFERENCE` —
   those normalize to camelCase keys and won't match the patterns below).
4. **Display name:** something stable, e.g. `aidp_demo_key` or
   `<workspace>_oci_signer`. This is what every tool's
   `conf.credential_name` points at.
5. **Four secret-key pairs:**

   | Key | Value |
   |---|---|
   | `tenancy` | the tenancy OCID `ocid1.tenancy.oc1..aaaa…` |
   | `user` | the user OCID for the service user above |
   | `fingerprint` | the fingerprint of the uploaded public key (`aa:bb:…`) |
   | `private_key` | **PEM body** of the private key, beginning `-----BEGIN PRIVATE KEY-----`, ending `-----END PRIVATE KEY-----`. Paste the contents, not a path. |

6. Save. Every tool that sets `conf.credential_name` to this display name
   now resolves it via `aidputils.secrets.get(name)` at invoke time.

### B. Database credential (for tools using oracledb / DBMS_CLOUD_AI)

Same flow, different keys:

| Key | Value |
|---|---|
| `username` | DB username |
| `password` | DB password |
| `connection_string` | TNS connect descriptor or DSN |
| `wallet_b64` | base64-encoded wallet zip body (optional, for mTLS-only ADB) |

For tools using `aidp_io.get_connection_data(catalog_key, ...)`, the catalog
binding already wires through credentials — you only need a separate
credential if you're connecting to a database that isn't bound to an AIDP
catalog.

### C. SMTP credential (for `email_toolkit`)

| Key | Value |
|---|---|
| `host` | e.g. `smtp.email.us-ashburn-1.oci.oraclecloud.com` |
| `port` | typically `587` |
| `username` | OCI SMTP credential username |
| `password` | OCI SMTP credential password |
| `from_address` | the `From:` to use |

### D. Third-party API bearer-token credential (for `web_toolkit`, generated REST tools)

| Key | Value |
|---|---|
| `api_key` | the bearer token / API key |
| `base_url` | (optional) the API base — keeps the tool config portable across envs |

## Tool-side pattern (what your `_execute_tool` does)

Every tool's `_execute_tool` follows the same three-step ritual:

```python
import aidputils.secrets as secrets
import oci

# 1. Resolve credential by display name (operator already created it).
credential_name = get_cfg(conf, "credential_name", "")
if not credential_name:
    return fail("conf.credential_name is required", "ValidationError")
bundle = secrets.get(credential_name)

# 2. Validate the bundle has every key this tool needs.
required = ("tenancy", "user", "fingerprint", "private_key")  # for OCI; varies by tool
missing = [k for k in required if not bundle.get(k)]
if missing:
    return fail(f"credential `{credential_name}` is missing keys: {missing}",
                "CredentialStoreError")

# 3. Use it. For OCI:
signer = oci.signer.Signer(
    tenancy=bundle["tenancy"], user=bundle["user"],
    fingerprint=bundle["fingerprint"],
    private_key_content=bundle["private_key"],
)
requests.get(url, auth=signer, timeout=timeout)
```

And always: **mask secret values in debug output and the returned
envelope**. Never log the PEM or the bearer token; truncate fingerprints
and OCIDs to first+last few characters.

## When you can leave `credential_name` empty

For internal AIDP calls that resource principal already covers, or for
tools that don't make external network calls at all. If you're unsure, set
it — the cost of pointing at a credential you don't end up needing is zero;
the cost of NOT pointing at one for a tool that needs it is a 401 in
production.
